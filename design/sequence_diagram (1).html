为什么采用异步方案？

方案 A：同步调用向量模型
QA 10ms
← QA 完成，等待向量返回 →
后续流程
向量生成 50ms（与 QA 同时开始）
QA 与向量并行，但 QA 需等待向量返回：实际耗时 = max(10, 50) = 50ms，整体时延增加 +40ms
方案 B：异步调用向量模型（当前方案）
QA 10ms
后续流程（召回 → 合并 → 排序）
向量生成 50ms（后台并行）
QA 阶段耗时 = 10ms，向量生成与召回并行，整体时延 +0ms
向量模型推理耗时约 50ms，而 QA 阶段本身仅需 10ms。方案 A 中虽然 QA 与向量生成并行执行，但 QA 需要同步等待向量返回才能继续，实际耗时取两者最大值 max(10, 50) = 50ms，相比原本 10ms 增加了 40ms。方案 B 采用异步调用，QA 阶段发出请求后立即返回，不等待向量结果，向量在后台与后续召回流程并行完成。排序阶段读取时向量通常已就绪；即使未就绪也可优雅降级，不影响可用性。

后续优化计划

Query 与 Item 推理 MEP 分离
将 Query 向量推理和 Item 向量推理拆分到独立的 MEP 实例中，解耦两条推理链路。Query 推理为在线实时路径，对延迟敏感；Item 推理可离线/近线批量执行，对吞吐量敏感。分离后可针对各自特性独立扩缩容和调优，互不影响。两条链路共享同一个向量模型，保证 Query 向量与 Item 向量处于同一语义空间，相似度计算结果一致。此外，模型推理耗时与输入文本长度正相关，部分 Item 文本远长于 Query，混合推理时长文本 Item 会拉高整体延迟并抢占计算资源。拆分后 Query 推理实例只处理短文本，推理延迟更低更稳定；Item 推理实例独立承担长文本负载，避免相互干扰，进一步优化在线链路的响应时间。
模型转换为 ONNX 格式
将向量模型从原始框架（PyTorch / TensorFlow）导出为 ONNX（Open Neural Network Exchange）格式。ONNX 是一种开放的跨框架模型表示标准，转换后具备以下优势：
跨平台兼容 — ONNX 解耦了训练框架与推理环境，同一模型可部署到 CPU、GPU、NPU 等不同硬件，无需针对每种平台重写推理代码。
推理加速 — ONNX Runtime 自动执行算子融合、内存优化、常量折叠等图级优化，在不修改模型的情况下即可获得显著的推理性能提升。
NPU 适配前提 — 华为 Ascend NPU 等硬件加速器的工具链（如 ATC / MindSpore Lite）通常以 ONNX 作为输入格式，转换为 ONNX 是部署到 NPU 的必经步骤。
标准化与可维护 — 模型以统一格式存储和版本管理，降低团队间的框架依赖差异，简化模型上线和回滚流程。
NPU 硬件加速推理
在 ONNX 转换基础上，进一步将模型部署到 NPU（Neural Processing Unit）。NPU 专为矩阵运算和张量计算设计，相比通用 CPU 在向量模型推理场景下可提供数倍的吞吐提升和更低的单次推理延迟，同时降低算力成本。最终目标是将向量生成延迟从当前的 50ms 压缩至 10ms 以内，为后续去除异步机制、实现全同步链路提供可能。
