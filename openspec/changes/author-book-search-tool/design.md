## Context

当前系统已经具备基础的Elasticsearch搜索能力，通过`ESClient`类提供连接和基本搜索功能。现有的`simple_slot_search.py`提供了基于match_phrase的槽位搜索，但缺乏针对作者-图书关系的专门搜索工具。测试数据`test_slots.csv`中已经包含了丰富的作者-图书对数据。

## Goals / Non-Goals

**Goals:**
- 实现支持模糊查询(fuzzy)和短语匹配(match_phrase)的作者-图书搜索工具
- 返回格式化的结果："作者名：xx，书名：xxx"
- 支持中英文混合查询和容错搜索
- 集成到现有的ReAct工具系统中
- 提供友好的错误处理和结果格式化

**Non-Goals:**
- 不重新实现Elasticsearch客户端基础功能
- 不修改现有的数据索引结构
- 不处理用户认证和权限控制

## Decisions

### 1. 工具位置和命名
- **位置**: `/tools/author_book_search.py`
- **函数名**: `author_book_search_tool`
- **装饰器**: 使用`@tool`装饰器保持一致性

### 2. 搜索策略设计
- **主要查询**: 使用`multi_match`查询同时搜索作者和书名字段
- **模糊查询**: 对中文作者名使用`fuzziness: "AUTO"`支持容错
- **短语匹配**: 对精确查询使用`match_phrase`提高准确性
- **权重设置**: 作者名权重高于书名(boost: 2.0 vs 1.0)

### 3. 结果格式化
- **成功结果**: 返回"作者名：{author}，书名：{book}"格式
- **多结果**: 使用换行分隔多个匹配结果
- **无结果**: 返回友好的提示信息
- **错误处理**: 捕获并返回具体的错误信息

### 4. 集成方式
- **独立工具**: 作为独立工具文件，不影响现有功能
- **索引配置**: 使用现有的`slots_index`索引
- **参数设计**: 接受查询字符串和可选索引参数

### 替代方案考虑
- **方案A**: 修改现有搜索工具 -  rejected, 会破坏现有功能
- **方案B**: 创建新工具 - accepted, 保持模块化和可维护性
- **方案C**: 使用外部API - rejected, 增加依赖和延迟

## Risks / Trade-offs

- **[数据质量风险]** → 依赖现有测试数据的完整性和准确性，建议增加数据验证
- **[性能风险]** → 模糊查询可能增加搜索时间，设置合理的size限制(默认10条)
- **[中文支持风险]** → Elasticsearch中文分词可能影响模糊匹配效果，需要测试验证
- **[错误处理风险]** → ES连接失败需要友好提示，避免暴露技术细节
